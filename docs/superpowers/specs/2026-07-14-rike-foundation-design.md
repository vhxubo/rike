# Rike 基础工程与组件骨架设计

## 1. 背景

Rike（中文名“日课”）是一款以手机端为主的每周学习计划应用。产品采用纸本日课式的线条视觉语言，后续需要支持周期预设、翻页动画、日期滑动、日/周/年视图缩放切换、PWA、本地持久化和数据备份迁移。

本规格仅覆盖第一阶段：建立可运行、可扩展、可验收的基础工程与组件骨架。第一阶段不实现真实学习计划业务、手势状态机、本地持久化、PWA 缓存策略或完整内容库。

## 2. 第一阶段目标

- 使用 React、Vite、TypeScript 和 Tailwind CSS 建立工程。
- 使用 `tailwind-variants` 管理基础组件变体。
- 纳入 Zustand、localForage、Motion 和 PWA 相关依赖，但不提前实现对应业务。
- 建立分层目录，隔离应用、页面、通用组件、领域功能、状态与存储适配器。
- 提供一个移动端优先的组件陈列首页，用于验收视觉、响应式布局和基础交互状态。
- 首页骨架展示当前日期和一条示例学习引导语，但不连接真实日期状态或完整内容库。
- 建立测试、静态检查、构建命令和持续交接文档约定。

## 3. 非目标

以下功能已进入总体需求，但不属于第一阶段实现范围：

- 周学习计划的创建、编辑、完成与周期预设。
- 左右滑动切换日期。
- 双指缩放或触控板缩放切换日、周、年视图。
- 翻页效果和完整的手势动画状态机。
- Zustand 状态持久化与 localForage 数据恢复。
- JSON 备份文件的实际导入、校验、迁移与导出。
- PWA manifest、Service Worker、离线缓存和安装体验。
- 400 条本地每日一句内容的生成、校审和正式接入。

## 4. 技术方案

### 4.1 工程基础

- React + Vite + TypeScript。
- Tailwind CSS 负责样式与设计令牌。
- `tailwind-variants` 负责组件尺寸、状态和视觉变体。
- Motion 提供后续动画基础；第一阶段只允许轻量入场和按压反馈。
- Zustand 作为后续应用状态容器。
- localForage 作为后续 IndexedDB 优先的本地存储适配层。
- PWA 使用 Vite 插件接入，但第一阶段不生成产品缓存策略。
- Vitest + Testing Library 负责组件测试。
- ESLint 负责静态检查。

### 4.2 目录边界

```text
src/
  app/                 应用入口、全局样式、Provider 与未来路由入口
  pages/               页面组合；第一阶段仅包含组件陈列页
  components/
    ui/                无业务基础组件
    layout/            应用壳与响应式导航
    paper/             纸本日课专属视觉组件
    feedback/          空状态和行内提示
  features/
    calendar/          日期游标、视图与手势接口
    backup/            备份数据契约与版本边界
    daily-guidance/    每日一句数据契约与筛选边界
  stores/              后续 Zustand store 入口
  lib/
    storage/           后续 localForage 适配器入口
  types/               跨模块共享类型
```

页面只组合功能与组件。通用组件不得直接读取 Zustand 或 localForage。功能模块不得把 localForage 的实现细节暴露给页面。

## 5. 组件设计

### 5.1 基础组件

- `Button`
- `IconButton`
- `TextField`
- `Textarea`
- `Checkbox`
- `Switch`
- `Badge`
- `Divider`
- `SegmentedControl`

组件需要支持明确的禁用、聚焦和按压状态。图标按钮使用现有图标库图标并提供可访问名称或 tooltip。

### 5.2 纸本组件

- `PaperSheet`：承载纸张背景、边框和稳定内容宽度。
- `RuledSection`：提供横线纸节奏，不承担业务布局。
- `BindingLine`：提供装订线或页边标记。
- `DateStamp`：展示紧凑日期信息。

### 5.3 布局组件

- `AppShell`
- `TopBar`
- `BottomNav`
- `PageContainer`

移动端采用单列纸页和底部导航。桌面端扩展内容宽度和留白，不建立独立的桌面组件体系。

### 5.4 反馈组件

- `EmptyState`
- `InlineNotice`

第一阶段只展示静态状态，不连接网络、存储或业务错误。

## 6. 视觉与动效

### 6.1 视觉方向

视觉基调为“纸本日课”，使用横线纸、装订线、墨色描边和克制的印章色。界面保持清晰、安静，不使用重度仿旧纹理或装饰性手账贴纸。

基础色包含：

- 宣纸白：主背景与纸页。
- 墨黑：主要文字和边线。
- 石墨灰：次要文字和禁用状态。
- 朱砂红：重点、日期印记和警示。
- 浅青绿：完成、平静和辅助状态。

使用系统中文字体栈，避免第一阶段引入外部字体网络依赖。卡片和控件使用小圆角，主要识别度来自 1px 线条、横线间距与页边结构。

### 6.2 动效边界

第一阶段允许：

- 页面内容轻量入场。
- 按钮和可交互控件的按压反馈。
- 尊重 `prefers-reduced-motion` 的降级。

翻页动画、左右滑动、双指缩放与日/周/年视图变形需要后续独立设计和实现。

## 7. 功能模块预留

### 7.1 日期与视图

`features/calendar` 预留以下契约：

```ts
type CalendarView = 'day' | 'week' | 'year'
```

后续日期状态至少包含当前日期游标和当前视图。左右滑动改变日期游标；缩放手势按离散层级在日、周、年之间切换。第一阶段只实现接收日期 props 的展示组件。

### 7.2 数据导入导出

备份格式采用 JSON，并包含：

- `schemaVersion`
- `exportedAt`
- 应用设置
- 学习计划
- 模板
- 周期预设
- 当前视图状态

PWA 缓存和其他可重新生成的数据不进入备份。后续导入流程必须先校验格式，再执行版本迁移，最后由用户确认是否覆盖现有数据。解析失败或版本不支持时不得破坏当前本地数据。

### 7.3 每日一句

每日一句是本地、学习相关的引导内容。目标用户为正在补习语言能力的初二女生，内容需要温和、具体、短句优先，并引导可完成的小步骤。

内容不得：

- 诊断或确认未经专业评估的心理疾病或人格状态。
- 反复提及意外、疾病或身份标签。
- 使用羞辱、威胁、内疚或依赖性表达。
- 把学习进度与个人价值绑定。

后续 400 条数据应按语言练习、阅读、写作、复习、休息、考试准备和自我观察等类别组织，并经过重复、语气、年龄适配和安全性校审。第一阶段只提供少量明确标记的示例内容。

## 8. 数据流与错误边界

第一阶段的数据流为 props 驱动：页面提供示例数据，展示组件只负责渲染和本地交互状态。

后续数据流遵循：

```text
页面/功能组件 -> Zustand action -> 领域状态 -> storage adapter -> localForage
```

存储适配器负责把底层错误转换为应用可理解的错误类型。导入操作在校验和迁移完成前不得写入正式状态。组件不得直接捕获或解释 localForage 错误。

## 9. 组件陈列首页

首页按以下区域展示：

1. 顶部日期与示例每日一句。
2. 基础控件。
3. 纸面组件。
4. 导航组件。
5. 空状态和提示状态。

手机端直接作为首屏验收。桌面端居中并限制阅读宽度。首页不伪装成已完成的周计划产品页，所有示例内容需要保持演示性质。

## 10. 测试与验收

第一阶段测试覆盖：

- 基础组件正常渲染。
- 关键尺寸、视觉与状态变体可选择。
- 表单标签、图标按钮名称和导航语义正确。
- 键盘可以到达并操作交互控件。
- `prefers-reduced-motion` 下不依赖动画传达必要信息。
- 组件陈列页在手机和桌面宽度下无明显溢出或重叠。

交付前必须通过：

```text
typecheck
lint
test
build
```
